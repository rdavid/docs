== History

A major part of Telco partner certification is the
{url-cnf-cert}[CNF Certification Test].
These test cases verify that the Telco partner's solution, deployed on Red Hat
OpenShift, follows the best practices.
The tests confirm that the implementation meets the recommended standards for
performance and compatibility.

One of the {url-catalog}[88 test cases] in the suite is
`platform-alteration-tainted-node-kernel`.
It verifies that the nodes hosting CNFs do not run tainted kernels.
For more on tainted kernels in Linux, see the {url-linux}[documentation].

While certifying OCP 4.10 and 4.11, our team hit difficulties with tainted
kernels.
Most failures came from incomplete kernel updates or outdated BIOS firmware.
Reboots and a BIOS firmware update resolved them.
For more, see this well-written article on the {url-solution}[issue].

== Story
The `platform-alteration-tainted-node-kernel` test in CNF Certification Tests
v4.2.4 fails consistently on OCP v4.12.10 with the following error message:
```
Taints mask=65536 - Decoded taints: [auxiliary taint, defined for and used by
distros (tainted bit 16)]
```

The test passes on OCP v4.11 and v4.13.

The test examines the kernel taint status on each node:
```
Pod IP: 192.168.26.51
If you don't see a command prompt, try pressing enter.
sh-4.4# chroot /host
sh-4.4# cat /proc/sys/kernel/tainted
65536
```

A zero value means the kernel is clean:
```
Pod IP: 192.168.26.52
If you don't see a command prompt, try pressing enter.
sh-4.4# chroot /host
sh-4.4# cat /proc/sys/kernel/tainted
0
```

The tainted kernel test keeps failing on one of the master nodes.
{url-dci-master-1}[The job] reports the failure on `master-1`; when I reboot
that master, {url-dci-master-2}[the failure] moves to the next one, `master-2`.
The failure migrates in a `0→>1→>2→>0` sequence after each reboot.

OCP 4.12 uses RHCOS based on 8.6.
From {url-solution}[here]:
```
With the TAINT_TECH_PREVIEW mask value being removed upstream and within RHEL 9
the TAINT_AUX bit is used instead of TAINT_TECH_PREVIEW within 8.6 and later
kernels.
```

The tainted node reports that it runs a technical preview module:
```
sh-4.4# dmesg|grep TECH
[17980.575402] TECH PREVIEW: kTLS may not be fully supported.
sh-4.4# uname -a
Linux master-0.r207-6node.r207.lab.eng.cert.redhat.com
4.18.0-372.49.1.el8_6.x86_64 #1 SMP Thu Mar 9 21:11:55 EST 2023 x86_64 x86_64
x86_64 GNU/Linux
```

An {url-bug}[existing bug] (Red Hat internal), about six months old, attributes
the issue to Metal3.
Its final resolution states that version 4.12 has no fix and that the taint
should clear with the move to version 4.13.
The Metal3 issue comes from building the container with RHEL 9-based images,
whose system calls load `kTLS`.
The {url-slack}[Slack thread] (Red Hat internal) covers the details.

== Problem

The certification team struggles to explain why the GA version of OCP 4.12 uses
`kTLS`, which is still in Technical Preview.
The situation concerns security experts, who could read it as a security
vulnerability.
The tests our partners run on their CNFs sharpen the need to address the issue
and give a clear explanation.

== Resolution

One option is to revert Metal3 to RHCOS8, which would drop `kTLS` from the GA
version, as intended.
The {url-metal3}[accompanying comment] (Red Hat internal) holds more on this
resolution.
